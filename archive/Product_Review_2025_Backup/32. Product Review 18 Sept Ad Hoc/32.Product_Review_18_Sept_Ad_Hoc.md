### 4. Тема: Детальные требования к доработке Strategic Planner Snapshot
- Ответственный: Новый разработчик, Сергей (менторство)
- Время обсуждения: 05:26 – 11:43
- Контекст / Цель: Глубокое обсуждение того, как должен выглядеть и функционировать snapshot (результат) стратегической оптимизации, чтобы он был полезным аналитическим инструментом.
- Ключевые моменты обсуждения и предложения команды:
  - Алексей уточнил, что сейчас в работе две задачи: read-only snapshot и добавление карты в сам планировщик.
  - Команда обсудила технические аспекты отображения большого количества маршрутов (100+) на карте, включая использование компрессированных треков и ограничение на одновременное отображение.
- Ключевые тезисы, директивы и мнения Дэна:
  - Read-only Snapshot: Новый разработчик должен сделать read-only view для результатов стратегического планирования. Этот экран должен выглядеть и функционировать один в один как Route List, но без возможности редактирования.
  - Карта в сценариях: После этого необходимо добавить полноценную карту непосредственно в интерфейс просмотра сценариев, чтобы можно было визуально оценивать locations.
  - Фильтрация и навигация: В snapshot критически не хватает фильтров по дню и неделе. Пользователь должен иметь возможность кликнуть на ячейку (например, "Неделя 32, День 1") и увидеть отфильтрованный Route List со всеми маршрутами за этот день. Также необходима навигация по дням с помощью клавиатуры.
  - Производительность карты: При отображении большого количества маршрутов система должна по умолчанию показывать упрощенный вид (только полилинии без маркеров), чтобы избежать проблем с производительностью.
- Решения и дальнейшие шаги:
  - Новый разработчик: Реализовать read-only snapshot для результатов Strategic Planner по образу и подобию Route List.
  - Фронтенд команда: Добавить в snapshot фильтры по дню/неделе.
  - Фронтенд команда: Реализовать логику для управления производительностью карты при большом количестве маршрутов.
- Итоговое резюме по теме:Были сформулированы четкие требования к доработке Strategic Planner. Основная цель — превратить snapshot из статического отчета в интерактивный аналитический инструмент, повторяющий проверенный и удобный интерфейс Route List, с мощной фильтрацией и производительной картой.
### 5. Тема: Проблема с качеством GPS-данных от клиента Amwaste и необходимость BigQuery
- Ответственный: Семён (анализ), Бэкенд команда
- Время обсуждения: 11:43 – 16:26
- Контекст / Цель: Семён представил результаты анализа качества GPS-треков от Amwaste, который выявил серьезные проблемы с данными от мобильных устройств, что мешает работе аналитических инструментов.
- Ключевые моменты обсуждения и предложения команды:
  - Большинство данных поступает с мобильных устройств (в основном iOS) и они очень редкие. Точки записываются только по прибытии на destination, что говорит о том, что приложение не работает в фоновом режиме.
  - Это известная проблема iOS, которая "убивает" фоновые процессы для экономии батареи.
- Ключевые тезисы, директивы и мнения Дэна:
  - Foreground Service: Дэн напомнил, что для решения этой проблемы был реализован Foreground Service, который позже отключили из-за жалоб. Он потребовал немедленно его вернуть, так как без постоянного трекинга данные бесполезны.
  - Аналитика использования: Необходимо начать собирать статистику по длительности сессий мобильного приложения. Это позволит доказать клиентам, что их водители сами закрывают приложение, и обосновать необходимость его постоянной работы.
  - Критическая директива: Все GPS-данные должны немедленно начать записываться в BigQuery. Это откроет огромные возможности для аналитики (построение heatmaps, анализ скорости и т.д.), которые сейчас заблокированы.
- Решения и дальнейшие шаги:
  - Мобильная команда: Повторно включить Foreground Service в iOS-приложении.
  - Бэкенд команда: Приоритезировать задачу по настройке экспорта всех GPS-треков в BigQuery.
  - Бэкенд команда: Реализовать сбор статистики по длительности сессий мобильного приложения.
- Итоговое резюме по теме:Проблема с качеством GPS-данных требует немедленных действий: возвращения Foreground Service и начала сбора статистики использования приложения. Стратегически принято ключевое решение о необходимости переноса всех GPS-данных в BigQuery, что является фундаментом для создания продвинутых аналитических инструментов.

В качестве немедленной меры команда попытается решить проблему с данными, а стратегически — принято решение о полной переработке этого сервиса силами основной команды для обеспечения надежности.
