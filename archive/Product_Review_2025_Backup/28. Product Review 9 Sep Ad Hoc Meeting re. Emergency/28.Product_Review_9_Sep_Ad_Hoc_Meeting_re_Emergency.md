### Отчет о стратегическом совещании по продукту
Дата: 9 сентября 2025 г.Участники: Дэн (CEO), Игорь Скринковский, Сергей, Вова и другие члены команды.Цель: Обсуждение требований нового потенциального клиента ("pizza wholesale"), определение критических функциональных пробелов, мешающих закрытию сделок, и выработка стратегии по их устранению в условиях ограниченного времени и ресурсов.
### 1. Тема: Конкурентная ситуация, финансовая срочность и требования клиента "Pizza Wholesale"
- Ответственный: Н/Д
- Время обсуждения: 01:24 – 04:35
- Контекст / Цель: Дэн обозначил критическую важность закрытия следующих 5-10 сделок в ближайшие 3 месяца из-за окончания финансового runway. Новый потенциальный клиент, "pizza wholesale", является стратегической возможностью, так как его сложные требования не может удовлетворить ключевой конкурент Descartes.
- Ключевые моменты обсуждения и предложения команды:
  - Команда обсудила, что у клиента есть потребности во всем спектре продвинутой функциональности, включая multi-day routing.
- Ключевые тезисы, директивы и мнения Дэна:
  - Финансовая срочность: У компании осталось 3 месяца до окончания средств. Необходимо "кровь из носа" закрыть следующие 5-10 сделок, чтобы показать прогресс инвесторам. Для этого пришлось сократить отдел продаж, поэтому вся нагрузка ложится на продукт.
  - Стратегическая важность клиента: Победа в сделке с "pizza wholesale" докажет превосходство продукта над Descartes, у которого отсутствует необходимый клиенту функционал.
  - Проблема ценообразования: Текущие клиенты платят недостаточно. Чтобы оправдать более высокие тарифы, продукт должен предоставлять продвинутые функции, которые сейчас не доведены до конца.
- Решения и дальнейшие шаги:
  - Команде необходимо сфокусироваться на завершении и полировке ключевых функций, которые требуются для закрытия ближайших сделок, в частности для "pizza wholesale".
- Итоговое резюме по теме:Встреча началась с установки на максимальную мобилизацию: финансовое положение компании требует немедленных результатов в виде новых продаж. Клиент "pizza wholesale" рассматривается как ключевая возможность продемонстрировать технологическое преимущество над конкурентами и оправдать более высокую стоимость продукта.
### 2. Тема: Обзор функциональных пробелов и приоритезация "Breaks"
- Ответственный: Игорь Скринковский (бэкенд), Вова (UI)
- Время обсуждения: 04:35 – 11:26
- Контекст / Цель: Анализ конкретных функциональных пробелов, выявленных в требованиях клиента "pizza wholesale", и определение приоритетов для их реализации.
- Ключевые моменты обсуждения и предложения команды:
  - Игорь Скринковский перечислил необходимые, но не полностью реализованные на фронтенде функции: mixit fleet, balancing (балансировка), skills (навыки), breaks (перерывы), pre/post trip insertion и multi-day routing.
  - Балансировка сейчас работает только по количеству остановок, но не по времени или дистанции.
  - Логика для breaks уже существует на уровне оптимизации (унаследована от задачи для FedEx), но отсутствует UI для их настройки в optimization profile.
- Ключевые тезисы, директивы и мнения Дэна:
  - Приоритет №1 — Breaks: Дэн категорично заявил, что отсутствие возможности настраивать перерывы на этапе оптимизации — самая болезненная проблема, из-за которой уходят клиенты. Эту задачу необходимо решить в первую очередь.
  - Требования к реализации: Правила для перерывов (break rules) не должны быть просто JSON-объектом. Это должна быть полноценная сущность в базе данных с уникальным ID. Такой подход позволит в будущем проводить аналитику (например, "показать все маршруты, где применялось правило перерыва X").
  - UI для Breaks: Необходимо создать UI для настройки правил перерывов в рамках optimization profile.
- Решения и дальнейшие шаги:
  - Вова: Подготовиться к завтрашнему дню, чтобы обсудить с Игорем и дорисовать UI для настройки breaks в новой версии optimization profile.
  - Игорь Скринковский / Бэкенд команда: Спроектировать и реализовать хранение правил для breaks в отдельной таблице в базе данных, а не в виде JSON.
  - Команда: Приоритезировать задачу по breaks как самую критичную для удержания и привлечения клиентов.
- Итоговое резюме по теме:Анализ требований клиента подтвердил наличие ряда пробелов в продукте. Дэн определил отсутствие настраиваемых перерывов (breaks) как наиболее критическую проблему, требующую немедленного решения. Команда сфокусируется на создании UI и доработке бэкенда для этой функции.
### 3. Тема: Новая концепция: Анализ исторических данных для Strategic Planner
- Ответственный: Игорь Скринковский (бэкенд), Вова (UI)
- Время обсуждения: 13:47 – 25:56
- Контекст / Цель: Дэн предложил новую прорывную идею — создать инструмент для импорта и анализа исторических данных о заказах клиента, чтобы автоматически генерировать входные данные для Strategic Planner. Это должно заменить дорогостоящие услуги консультантов и создать мощный "вау-эффект" на демо.
- Ключевые моменты обсуждения и предложения команды:
  - Команда уточнила, что речь идет о создании "зеркала" структуры ордеров, но предназначенного исключительно для аналитики, чтобы не засорять основную транзакционную базу данных.
  - Было предложено использовать для этой цели дешевое и быстрое хранилище, такое как BigQuery.
- Ключевые тезисы, директивы и мнения Дэна:
  - Проблема: Сейчас для подготовки данных к стратегическому планированию клиенты нанимают дорогих консультантов (вроде Паркера), которые неделями анализируют данные в Excel.
  - Решение: Создать инструмент, куда можно загрузить файл с историческими данными клиента за год (например, 1 миллион заказов). Система должна за 10-20 секунд обработать эти данные и выдать готовую аналитику: частоту посещений, предпочтительные дни и временные окна для каждого клиента.
  - "Wow-эффект" для продаж: Возможность обработать огромный массив данных на лету и сразу же построить на его основе качественный стратегический план станет убойным аргументом на демо.
  - Техническое требование: Для быстрой обработки необходимо требовать от клиентов предоставления координат в файле, чтобы избежать медленного геокодирования.
  - Интеграция со Strategic Planner: Результаты анализа (хитмэпы, гистограммы, статистика) должны быть не просто отчетом, а напрямую использоваться как входные данные для Strategic Planner.
- Решения и дальнейшие шаги:
  - Игорь Скринковский / Бэкенд команда: Спроектировать архитектуру для импорта и хранения исторических данных в BigQuery.
  - Артем / Бэкенд команда: Подготовить SQL-запросы для анализа данных в BigQuery.
  - Вова / Фронтенд команда: Спроектировать UI для нового инструмента, включая процесс импорта и визуализацию результатов (хитмэпы, статистика).
  - Разработать механизм передачи результатов анализа в Strategic Planner.
- Итоговое резюме по теме:Была предложена и принята концепция нового мощного аналитического инструмента, который автоматизирует подготовку данных для стратегического планирования. Это не только решит серьезную проблему для клиентов, но и станет уникальным торговым преимуществом, способным кардинально изменить процесс продаж.
### 4. Тема: UI/UX и функциональность Dispatch Board
- Ответственный: Сергей (фронтенд), Вова (UI)
- Время обсуждения: 47:46 – 57:50, 1:03:40 - 1:12:49
- Контекст / Цель: Обсуждение текущего состояния и видения Dispatch Board как ключевого инструмента для ручного и полуавтоматического распределения заказов.
- Ключевые моменты обсуждения и предложения команды:
  - Команда подтвердила, что на старой карте (Routes Map) существует базовая, но глючная функциональность перетаскивания остановок между маршрутами.
  - Сергей отметил, что Dispatch Board в текущем виде недоделан.
  - Вова предложил интегрировать Dispatch Board непосредственно в Routes List: добавить панель с нераспределенными ордерами, откуда их можно будет перетаскивать на таймлайн конкретного маршрута.
- Ключевые тезисы, директивы и мнения Дэна:
  - Дэн согласился с видением Вовы, назвав его "шикарным" решением. Dispatch Board не должен быть отдельной страницей, а должен быть интегрирован с Routes List.
  - Он подчеркнул важность интерактивности: карта на этом экране должна быть "контекстной" — при наведении на водителя показывать его маршрут, при наведении на заказ — его местоположение и ближайшие маршруты.
- Решения и дальнейшие шаги:
  - Вова / Сергей: Спроектировать и реализовать Dispatch Board как расширение Routes List, добавив панель с нераспределенными заказами и drag-and-drop на таймлайн.
  - Проработать и внедрить контекстную карту для этого экрана.
- Итоговое резюме по теме:Принято решение отказаться от разработки Dispatch Board как отдельного экрана. Вместо этого будет создан интегрированный инструмент внутри Routes List, объединяющий список маршрутов, таймлайн и панель нераспределенных заказов, что обеспечит интуитивно понятный и эффективный процесс диспетчеризации.
### 5. Тема: Сезонность и гибкость расписаний для Customer Location
- Ответственный: Вова (UI)
- Время обсуждения: 58:18 – 1:03:40
- Контекст / Цель: Обсуждение необходимости сделать редактор расписаний (графиков работы и временных окон) для Customer Location более гибким, добавив поддержку сезонности.
- Ключевые моменты обсуждения и предложения команды:
  - Вова подтвердил, что у него уже есть наработки (пресеты для расписаний), которые можно применять к нескольким клиентам. Он предложил добавить к этим пресетам даты начала и окончания действия (effective dates).
- Ключевые тезисы, директивы и мнения Дэна:
  - Дэн указал на потребность клиентов задавать разные расписания для разных сезонов (например, зимой магазин закрывается раньше).
  - Обсуждались два подхода: выбор сезона (зима/лето) или выбор конкретных месяцев. Дэн склонился к выбору месяцев, так как это дает больше контроля.
  - Идея Вовы с пресетами и датами действия была одобрена как хорошее, гибкое решение.
- Решения и дальнейшие шаги:
  - Вова: Дорисовать макеты для редактора расписаний, включив в них возможность создавать несколько расписаний для одной локации, каждое из которых имеет свой период действия (даты начала и окончания).
  - В качестве демонстрации для клиента можно создать "нерабочую" версию этого UI на стейджинге, используя статический JSON.
- Итоговое резюме по теме:Для удовлетворения реальных потребностей клиентов решено усовершенствовать редактор расписаний. Будет добавлена возможность задавать несколько временных графиков для одной локации с указанием периода их действия, что позволит легко управлять сезонными изменениями в работе.
